# Fix: SQL Injection Vectors

This document explains **what SQL injection is**, **where it appeared in the codebase**, and **how we fixed it** without changing behaviour.

---

## 1. What is SQL injection?

SQL is built as **text**. If part of that text comes from **user input** (or from data that could ever be influenced by an attacker) and is **concatenated** into the string instead of passed as a **parameter**, an attacker can break out of the intended SQL and run their own.

**Example (values, not identifiers):**

```sql
-- Intended: look up user by id
SELECT * FROM users WHERE id = $1   -- safe: $1 is a parameter

-- Dangerous: concatenating user input
SELECT * FROM users WHERE id = '" + req.params.id + "'  -- if id is "1; DROP TABLE users; --" you run that too
```

**Defence for values:** use **parameterized queries**: `pool.query('SELECT ... WHERE id = $1', [id])`. The driver sends the value separately; it is never parsed as SQL.

**Identifiers (table/schema/column names)** cannot be parameterized in standard SQL. So we must **never** build them from arbitrary input. We either:

- use a **whitelist**: only allow a fixed set of known identifiers (e.g. column names from our model), or  
- validate with a **strict pattern** (e.g. schema names must match `establishment_[a-zA-Z0-9_]+`) and then interpolate only after the check.

---

## 2. What was wrong in our code?

### 2.1 Schema name interpolated into SQL (establishment.ts)

```ts
const ordersResult = await pool.query(`
  SELECT ... FROM "${establishment.schema_name}".orders ...
`);
```

`schema_name` comes from the **database** (establishments table). If that value were ever wrong or malicious (bug, migration, or compromise), it would be pasted directly into the SQL. For example:

- `schema_name = 'x"; DELETE FROM users; --'`  
  would produce: `FROM "x"; DELETE FROM users; --".orders` and execute the `DELETE`.

So **any** identifier we interpolate must be validated before use. Here we know schema names are always generated by us as `establishment_<uuid>` or `establishment_<id>`, so we validate that format and **reject** anything else before building the query.

### 2.2 Dynamic column names in UPDATE (orderModel.ts, productModel.ts)

```ts
const fields = Object.keys(order).filter(key => key !== 'id' && key !== 'establishment_id');
const setClause = fields.map((field, index) => `${field} = $${index + 3}`).join(', ');
const query = `UPDATE orders SET ${setClause} ...`;
```

Here **column names** come from the **keys of the object** passed in. If a caller (or a bug) passes `{ "total_amount; DROP TABLE orders; --": 0 }`, that key is interpolated into the SQL and can execute arbitrary statements.

So we must **not** trust object keys as column names. We use a **whitelist**: only allow column names that are real, updatable columns of the table (e.g. from our TypeScript types). We filter to that list before building the SET clause.

---

## 3. How we fixed it

### 3.1 Schema name (establishment.ts)

- Added a **validation** that `schema_name` matches the pattern we use in the app:  
  `^establishment_[a-zA-Z0-9_]{1,64}$`
- Before using `establishment.schema_name` in a query, we run this validation. If it fails, we throw and do not run the query.
- After validation, the only characters in the name are letters, digits, and underscores, so it is safe to quote and interpolate in PostgreSQL (no quote or semicolon can appear inside the identifier).

So we still interpolate the schema name (because PostgreSQL doesn’t support parameters for identifiers), but we **only** do it after a strict format check.

### 3.2 Order UPDATE (orderModel.ts)

- Defined an explicit **whitelist** of allowed column names for updates:  
  `total_amount`, `total_tax`, `payment_method`, `status`, `notes`, `tips`, `change`.
- When building the UPDATE, we **filter** `Object.keys(order)` to only keys that are in this whitelist. Only those columns are used in the SET clause.
- Values are still passed as parameters (`$1`, `$2`, …). Only the **identifiers** (column names) are built from the whitelist.

So no user- or caller-controlled string can become a column name; only our fixed list can.

### 3.3 Product UPDATE (productModel.ts)

- Same idea: a **whitelist** of allowed product columns for update:  
  `name`, `price`, `tax_rate`, `category_id`, `happy_hour_discount_percent`, `happy_hour_discount_fixed`, `is_happy_hour_eligible`, `is_active`.
- We only build the SET clause from keys that are in this list. Values stay parameterized.

Result: no dynamic, unvalidated identifier is ever interpolated into SQL; schema names are validated by format, and column names are restricted to a whitelist.
